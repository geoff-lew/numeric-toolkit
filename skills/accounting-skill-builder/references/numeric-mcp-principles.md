# Numeric MCP Principles

How automations should interact with Numeric. These principles are the craft any agent that touches Numeric should follow. They keep automations safe, observable, and self-bootstrapping across cycles.

## Task descriptions are the persistence layer

Numeric tasks have descriptions that survive across periods. Use them.

When an automation has user preferences (vendor overrides, exclusions, materiality thresholds, criteria), write them into the task description as a structured section:

```
## Accrual preferences

vendor_overrides:
  - acme_inc: use_prior_month
vendor_exclusions:
  - one_time_consultant_inc
criteria_overrides:
  min_threshold: 5000
```

Next cycle's run reads the prior period's task description, loads the preferences, and carries them forward. The automation self-bootstraps — no separate config file, no database.

Precedence order: defaults < prior-period preferences < current task description's explicit instructions < explicit user override during the run.

## Comments are the audit trail

Every automation that touches a task posts a comment via `add_task_comment` summarizing what it did:

```
Auto-generated by [skill-name] on 2026-04-30 14:32 UTC.
- Pulled 7 months of transactions for Software expense
- Identified 8 accrual candidates above $5K threshold
- Generated workpaper at outputs/.../workpaper.xlsx
- Excluded 2 vendors per task description preferences
```

Comments serve two purposes: audit trail for the controller, and breadcrumbs for the next run to detect "this period was already processed."

## Cold-start sequence

Every Numeric-touching automation starts with the same sequence:

1. `list_workspaces` (only if workspace not in memory)
2. `set_workspace`
3. `get_workspace_context` — note open period, current user, period_id, entities, holidays
4. Cache the context to `outputs/.numeric_session_cache/{workspace_id}.json` (24h TTL)
5. Parallel reads as needed: `list_reports`, `list_financial_accounts`, `list_tasks` with server-side filters

Don't re-pull cached data within a session.

## Filter at the API surface

`list_tasks` accepts `status`, `task_type`, `assignee_id`, `report_ids`, `name_contains`, `period_id`, `include_description`. Push every filterable field into the call. The MCP applies filters server-side and returns a smaller payload. Only filter in-script what the API can't handle.

Don't pull all tasks then filter — slow and wastes context.

## Submission is gated

Automations do not call `submit_task` without explicit user confirmation. Workflow:

1. Build the artifact (workpaper, JE, commentary, etc.)
2. Validate it
3. Surface the artifact, validation results, and the planned task action
4. `AskUserQuestion`: "Submit task / download for manual review / re-run with changes"
5. Submit only on explicit "submit"

Same rule applies to `update_flux_explanation`, `edit_task` (when changing critical fields), and any mutation that lands in production.

## Reading flux explanations

Use `get_flux_explanations` to check existing content before posting. If content exists in the current period, append with `---` divider. Never overwrite. Append-with-divider preserves prior preparer work and audit history.

For tone reference, read the prior period's explanation on the same row — keeps drafting voice consistent across months.

## Conclusion-first writing

Anything posted to Numeric (flux explanations, task comments, audit notes) opens with the conclusion in one sentence: total variance, primary driver, concentration. Bullets follow. Mirrors how the toolkit's `automatically-draft-flux-explanations` skill writes.

## Period awareness

Resolve the open period from `get_workspace_context` rather than assuming. Check `period_id` exists before any period-bound call. If a user says "this close" while a period is mid-flight, ask whether to include in-flight or wait for completion.

Never write to a locked period without explicit user confirmation.

## Workspace-tier sizing

Workspace size determines reasonable defaults:
- Small (<5 entities): tight materiality thresholds, lookback windows can be wider, fan-out probably not needed
- Medium (5–10 entities): 5× materiality, fan out across entities
- Large (>10 entities): 10× materiality, batch fan-out at 3 parallel calls

The `get_workspace_context` response gives entity count. Use it to set defaults.

## Reusing existing toolkit skills

Before designing a new automation that touches Numeric, check `references/skill-catalog.md`. If an existing skill covers the workflow, the recommendation is to refine it (different trigger criteria, materiality, entity scope) rather than greenfield. Refinement options the Numeric Integration agent considers:
- Wrap the existing skill in a scheduled task
- Adjust default thresholds or windows for the user's workspace tier
- Layer an extra control or output format
- Compose two existing skills (e.g., close-pulse → close-retro)

Greenfield only when no existing skill covers the workflow's spine.
